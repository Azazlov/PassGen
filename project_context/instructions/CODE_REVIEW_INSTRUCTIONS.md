## 1. НАЗНАЧЕНИЕ

Эта инструкция описывает процесс проведения код-ревью для проекта PassGen, включая проверку соответствия ТЗ, код-стайлу и лучшим практикам.

---

## 2. ТИПЫ РЕВЬЮ

### 2.1 Полное ревью (CODE_REVIEW.md)

**Расположение:** `project_context/reviews/CODE_REVIEW_YYYY-MM-DD.md`

**Назначение:** Комплексная проверка компонента/проекта

**Когда проводить:**
- После завершения этапа
- Перед релизом
- По запросу

---

### 2.2 Тематическое ревью (TOPIC_REVIEW.md)

**Расположение:** `project_context/reviews/TOPIC_REVIEW.md`

**Назначение:** Проверка конкретного аспекта

**Примеры:**
- `UI_UX_CODE_REVIEW.md` — ревью UI/UX
- `SECURITY_REVIEW.md` — ревью безопасности
- `PERFORMANCE_REVIEW.md` — ревью производительности

---

### 2.3 Быстрое ревью (PULL_REQUEST_REVIEW.md)

**Расположение:** `project_context/reviews/PR_N.md`

**Назначение:** Проверка pull request

**Когда проводить:**
- Перед слиянием веток
- При изменении критического кода

### 3.4 Оценка

**Формула расчёта:**
```
Общая оценка = Σ(Вес_категории * Оценка_категории) / ΣВесов
```

**Шкала оценок:**
- ✅ 90-100% — Отлично
- ⚠️ 70-89% — Хорошо (требует улучшений)
- ❌ < 70% — Плохо (критичные проблемы)

### 4.2 Код-стайл

**Правила именования:**
```dart
// ✅ Хорошо
class PasswordEntry { }  // PascalCase
final String serviceName = '...';  // camelCase
const int maxAttempts = 5;  // camelCase для const
enum AppTab { }  // PascalCase

// ❌ Плохо
class passwordEntry { }  // Не PascalCase
final String ServiceName = '...';  // Не camelCase
```

**Форматирование:**
```dart
// ✅ Хорошо
class AuthController extends ChangeNotifier {
  final String pin;  // поля в начале
  int attempts;
  
  AuthController(this.pin);  // конструктор
  
  void verify() { }  // методы после
}

// ❌ Плохо
class AuthController{  // Нет пробела перед {
  int attempts;final String pin;  // Всё в одну строку
  void verify(){}  // Нет пробела перед {
}
```

## 8. ЧАСТЫЕ ПРОБЛЕМЫ

### 8.1 Архитектурные

**Проблема:** Нарушение Clean Architecture
```dart
// ❌ Presentation зависит от Data
import 'package:sqflite/sqflite.dart';

class GeneratorScreen extends StatelessWidget {
  final Database db;  // Прямая зависимость от БД
}

// ✅ Presentation зависит от Domain
import '../../domain/usecases/generate_usecase.dart';

class GeneratorScreen extends StatelessWidget {
  final GenerateUseCase useCase;  // Зависимость от UseCase
}
```

---

### 8.2 Безопасность

**Проблема:** Хардкод секретов
```dart
// ❌ Пароль в коде
const String apiKey = 'super_secret_key';

// ✅ Переменные окружения
final String apiKey = const String.fromEnvironment('API_KEY');
```

---

### 8.3 Производительность

**Проблема:** Тяжёлые операции в UI потоке
```dart
// ❌ Блокировка UI
void loadData() {
  final data = database.query();  // Синхронно
  setState(() { ... });
}

// ✅ Асинхронно
Future<void> loadData() async {
  final data = await database.query();  // Асинхронно
  setState(() { ... });
}
```

## 9. МЕТОДОЛОГИИ

### 9.1 SMART критерии

Проблемы должны быть:
- **S**pecific — Конкретные
- **M**easurable — Измеримые
- **A**chievable — Достижимые
- **R**elevant — Релевантные
- **T**ime-bound — Ограниченные по времени

---

### 9.2 Метод 4 глаз

**Принцип:**
```
Минимум 2 ревьюера должны одобрить изменения
```

**Процесс:**
1. Автор создаёт PR
2. Ревьюер 1 проверяет
3. Ревьюер 2 проверяет
4. При одобрении — слияние
