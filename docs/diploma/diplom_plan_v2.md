## 1. Позиционирование системы (зафиксировано после интервью)

### 1.1. Итоговое позиционирование

**PassGen** — кроссплатформенный менеджер паролей для малого бизнеса и фрилансеров с локальным шифрованием без использования облачной инфраструктуры.

### 1.2. Ключевые характеристики (что реально в коде на момент плана)

| Параметр | Значение |
|---|---|
| Язык | Dart ≥ 3.9 |
| Фреймворк | Flutter ≥ 3.24 |
| СУБД | SQLite (`sqflite` + `sqflite_common_ffi`) |
| Архитектура | Clean Architecture (presentation / domain / data + core + shared) |
| Шифрование | ChaCha20-Poly1305 (AEAD) |
| Деривация ключа | PBKDF2-HMAC-SHA256, v2 = 600 000 итераций (соответствует OWASP 2024) |
| Генератор случайных чисел | `Random.secure()` (CSPRNG) |
| Платформы (по факту) | Windows, Linux, macOS, Android, iOS, Web (6 платформ — сборка; тестирование — см. Глава 3) |
| Объём кода | ~15 700 строк Dart в `lib/`, ~5 800 строк тестов |
| Тестирование | 165 модульных тестов (`mockito`, `flutter_test`) |
| Состояние защиты | PIN-код (PBKDF2 + constant-time сравнение + блокировка после 5 попыток) |

### 1.3. Скоп доработок в коде **до защиты**

Следующие функции должны быть реализованы до подачи на защиту:

1. **Биометрическая аутентификация** через пакет `local_auth` (отпечаток / лицо) — как второй фактор к PIN.
2. **Многопрофильность** — несколько учётных записей пользователей на одном устройстве без сервера (одна SQLite-база, таблица `profiles`, внешний ключ `profile_id` в связанных таблицах).
3. **Глитчирование текста** — детерминированное преобразование введённой пользователем осмысленной последовательности в стойкий пароль (смещения, подстановка визуально схожих символов, добавление солевых элементов).
4. **QR-обмен** записями паролей — экспорт отдельной записи в QR-код с расшифровкой на принимающем устройстве (`qr_flutter` + `mobile_scanner`).
5. **Повышение `lockoutDurationSeconds`** с текущих 30 сек до разумного значения (рекомендуется прогрессивная задержка или ≥ 30 мин).
6. **Замеры производительности** (PBKDF2, генерация, шифрование) на 4 платформах: Linux, Windows, macOS, Android-эмулятор (физический Android недоступен).

### 1.4. Скоп, перенесённый в «Перспективы развития»

- Серверная часть с ролевой моделью администратор/сотрудник.
- Двухфакторная аутентификация через TOTP (Google Authenticator, Яндекс Ключ) как второй фактор к PIN.
- Поддержка отечественных криптоалгоритмов (ГОСТ Р 34.12-2018 «Кузнечик») через сертифицированные криптопровайдеры (например, КриптоПро CSP) — с учётом того, что аудитированных Dart-реализаций на текущий момент нет.
- Синхронизация по локальной сети между устройствами (совместимый вариант — экспорт в приложение LocalSend).
- Плагин для браузеров.
- Интеграция с системными менеджерами паролей ОС (GNOME Keyring, Windows Credential Manager, macOS Keychain).
- Разделение на нативные UI-варианты (Cupertino для iOS, Material You для Android) при появлении в экосистеме Flutter разделения UI и бизнес-логики с сохранением общего domain-слоя.

## 2. Структура работы (по главам)

### 2.1. Введение (1,5–2 стр)

**2.1.1. Актуальность темы.**
Рост числа онлайн-сервисов увеличивает количество учётных записей у каждого пользователя и малого бизнеса: по открытым данным отраслевых исследований, у среднего пользователя более 100 различных учётных записей. Запомнить надёжные уникальные пароли становится практически невозможно, а повторное использование одного пароля на нескольких сервисах приводит к массовым компрометациям при утечках.

Облачные менеджеры паролей (LastPass, 1Password) неоднократно подвергались атакам, при которых похищались зашифрованные хранилища мастер-ключей пользователей. В условиях санкционного давления, ограничения доступа к зарубежным облачным сервисам и требований российского законодательства о персональных данных (152-ФЗ) актуальным становится создание **локального** менеджера паролей, работающего без облачной инфраструктуры, с открытым исходным кодом и сильной криптографической защитой. Такое решение особенно востребовано у малого бизнеса и фрилансеров, не имеющих бюджета на корпоративные системы единой точки входа.

**2.1.2. Цель работы.**
Разработать кроссплатформенное приложение для безопасного локального хранения и генерации паролей с использованием современных криптографических алгоритмов.

**2.1.3. Задачи исследования:**
1. Проанализировать предметную область управления паролями и существующие решения (LastPass, KeePassXC, Bitwarden, Яндекс Ключ).
2. Сформулировать функциональные и нефункциональные требования к разрабатываемой системе.
3. Выбрать и обосновать технологический стек: язык программирования, фреймворк, СУБД, криптографические алгоритмы.
4. Спроектировать архитектуру приложения на основе Clean Architecture и схему базы данных SQLite.
5. Реализовать модуль криптографической защиты (ChaCha20-Poly1305, PBKDF2-HMAC-SHA256) и модуль аутентификации (PIN + биометрия).
6. Разработать пользовательский интерфейс и реализовать основные сценарии: генерация паролей, хранение, импорт/экспорт, многопрофильность, QR-обмен записями.
7. Провести функциональное и security-тестирование, подготовить пользовательскую документацию.

**2.1.4. Объект исследования.**
Процессы защищённого хранения и генерации аутентификационных данных пользователей на локальных устройствах.

**2.1.5. Предмет исследования.**
Методы и программные средства построения кроссплатформенного менеджера паролей с локальным шифрованием данных.

**2.1.6. Методы исследования:**
- Анализ предметной области и сравнительный анализ существующих решений.
- Моделирование с использованием UML (Use Case, Class, Sequence, Activity, Component, Deployment), IDEF0 для бизнес-процессов, IDEF1X для ER-моделирования базы данных.
- Методы проектирования ПО: Clean Architecture, принципы SOLID, шаблон Repository.
- Прикладные методы: программирование на Dart/Flutter, функциональное и модульное тестирование (Mockito, flutter_test).

- Экспертная оценка производительности криптографических операций и генерации паролей (замеры на реальных устройствах).

**2.1.7. Практическая значимость.**
Разработанное приложение может быть использовано индивидуальными пользователями, фрилансерами и сотрудниками малого бизнеса для безопасного хранения паролей без передачи данных в облачные сервисы, что снижает риски утечек и обеспечивает контроль над собственными данными.

### 2.2. Глава 1. Анализ предметной области и постановка задачи (8–10 стр)

**2.2.1. Общая характеристика предметной области.**
Типовой пользователь рассматриваемой системы — малый бизнес (1–20 сотрудников, без выделенного IT-отдела) и индивидуальные пользователи / фрилансеры. Среднее число учётных записей на сотрудника — 50–150. Основные требования заказчика: отсутствие ежемесячной подписки, хранение данных на собственных устройствах, удобство использования, работа на всех основных платформах, поддержка русского языка.

**2.2.2. Анализ бизнес-процессов.**
Рассматриваются четыре ключевых процесса, представленные в нотации **IDEF0** (диаграммы в Приложении В):

- **Процесс 1: Создание нового пароля для сервиса.**
  *AS-IS:* пользователь придумывает пароль в уме или записывает в блокнот → высокий риск слабого или повторного пароля.
  *TO-BE:* генератор PassGen создаёт пароль по заданным параметрам → гарантированно уникальный и стойкий.

- **Процесс 2: Поиск и использование существующего пароля.**
  *AS-IS:* пользователь ищет запись в блокноте или текстовом файле → ошибки, риск утечки.
  *TO-BE:* поиск и фильтрация по категориям в PassGen → копирование в буфер обмена с автоматической очисткой через 60 секунд.

- **Процесс 3: Смена пароля с сохранением истории.**
  *AS-IS:* при смене пароля старое значение теряется, при компрометации невозможно отследить момент компрометации.
  *TO-BE:* таблица `password_history` хранит все версии с отметками времени.

- **Процесс 4: Обмен паролями между устройствами.**
  *AS-IS:* передача через мессенджер / email в открытом виде.
  *TO-BE:* экспорт записи в QR-код, сканирование на другом устройстве; либо экспорт всей базы в зашифрованный файл `.passgen` и передача через сторонний инструмент (например, LocalSend).

**2.2.3. Анализ существующих решений.**
Проводится сравнительный анализ четырёх аналогов (подробная таблица в тексте главы):

| Аналог | Тип | Модель хранения | Ключевые особенности |
|---|---|---|---|
| LastPass | Коммерческий, облачный | Облако | Синхронизация, но утечки 2022 г. |
| KeePassXC | Открытый код, локальный | Файл `.kdbx` | Бесплатный, но устаревший интерфейс и ограниченные настройки генератора |
| Bitwarden | Открытый код, облако / self-host | По умолчанию облако | Self-host требует поднятия сервера |
| Яндекс Ключ | Российский, облачный | Облако Яндекса | Привязка к экосистеме Яндекса |

Критерии сравнения: открытость кода, модель хранения, используемые алгоритмы шифрования, кроссплатформенность, стоимость, доступность в РФ, соответствие 152-ФЗ, удобство интерфейса, гибкость настроек генератора.

**2.2.4. Выявленные проблемы.**

1. Зависимость от иностранных облачных сервисов: санкционные риски, блокировки, невозможность оплаты.
2. Утечки мастер-ключей у крупных облачных провайдеров (LastPass 2022).
3. Риск нарушения требований 152-ФЗ при хранении персональных данных в зарубежных облаках.
4. Сложность и устаревший интерфейс локальных open-source решений (KeePassXC).
5. Ограниченная гибкость настроек генератора в большинстве аналогов.
6. Отсутствие на рынке решения, объединяющего открытый исходный код, локальное хранение, современный интерфейс Material 3 и кроссплатформенность из одного кода.

**2.2.5. Требования к разрабатываемой системе.**

*Функциональные (ФТ):*
- ФТ-1. Аутентификация по PIN-коду с хэшированием PBKDF2-HMAC-SHA256.
- ФТ-2. Биометрическая аутентификация (отпечаток / распознавание лица) как второй фактор.
- ФТ-3. Многопрофильность — поддержка нескольких учётных записей на одном устройстве.
- ФТ-4. Генерация паролей с настраиваемыми параметрами (длина, наборы символов, уникальность, исключаемые символы).
- ФТ-5. Хранение паролей с метаданными: сервис, логин, категория, даты создания и обновления.
- ФТ-6. История изменений паролей (все версии каждой записи).
- ФТ-7. Поиск и фильтрация записей по сервису, логину, категории.
- ФТ-8. Импорт и экспорт данных в зашифрованном формате `.passgen`.
- ФТ-9. Обмен отдельными записями через QR-код между устройствами.
- ФТ-10. Глитчирование пользовательского текста в стойкий пароль.
- ФТ-11. Уведомления о слабых, устаревших и дублирующихся паролях.
- ФТ-12. Журнал событий безопасности (авторизации, экспорт, смена PIN и т.д.).

*Нефункциональные (НФТ):*
- НФТ-1. Кроссплатформенность: Windows, Linux, macOS, Android, iOS, Web из одного кодобаза.
- НФТ-2. Шифрование всех хранимых паролей алгоритмом ChaCha20-Poly1305 (AEAD).
- НФТ-3. Время отклика интерфейса ≤ 200 мс при количестве записей до 10 000.
- НФТ-4. Соответствие принципам OWASP Mobile Top 10 и OWASP ASVS.
- НФТ-5. Полная автономность (отсутствие сетевых запросов).
- НФТ-6. Открытый исходный код под лицензией MIT.
- НФТ-7. Локализация интерфейса на русский язык.
- НФТ-8. Минимизация англицизмов в терминологии.

**2.2.6. Моделирование процессов.**
В Главе 1 приводятся краткие сводные диаграммы; полные версии — в Приложениях Б и В:

- **Use Case-диаграмма** (UML) — роли и сценарии использования.
- **IDEF0-диаграммы** четырёх основных бизнес-процессов (см. п. 2.2.2).

**2.2.7. Соответствие требованиям 152-ФЗ.**
Короткий абзац: PassGen не является оператором персональных данных, так как:
- не передаёт данные третьим лицам;
- не обрабатывает их в автоматизированной информационной системе персональных данных;
- всё хранение осуществляется локально на устройстве пользователя, в зашифрованном виде.

Следовательно, применение PassGen не обязывает организацию регистрироваться в Роскомнадзоре как оператор ПДн.

**2.3.5. Выбор криптографической библиотеки.**
Сравниваются: `cryptography` (pub.dev), `pointycastle`, встроенный `dart:crypto`. Итоговый выбор: **`cryptography`** (реализация ChaCha20-Poly1305, PBKDF2), дополнительно `crypto` для хэш-функций.

**2.3.6. Выбор криптоалгоритмов. Сравнение с ГОСТ «Кузнечик».**
Рассматриваются варианты:
- **ChaCha20-Poly1305** (RFC 8439) — AEAD-шифр, выбран для реализации.
- **AES-256-GCM** — отраслевой стандарт, но требует аппаратной поддержки для эффективности.
- **ГОСТ Р 34.12-2018 «Кузнечик»** — отечественный стандарт.

Обоснование выбора ChaCha20-Poly1305 над Кузнечиком:
1. Производительность на устройствах без аппаратного ускорения AES/Кузнечика (ChaCha20 изначально software-friendly).
2. Наличие аудитированной Dart-реализации; для Кузнечика пока нет независимо аудитированного порта на Dart.
3. Совместимость с международными протоколами (TLS 1.3, WireGuard, Signal).
4. Сертификация алгоритма ГОСТ обязательна только при обработке сведений, составляющих государственную тайну, или персональных данных высшего уровня защищённости — что не относится к рассматриваемой области применения.

Поддержка ГОСТ «Кузнечик» вынесена в «Перспективы развития» через сертифицированные криптопровайдеры при использовании в государственных организациях.

**2.3.7. Дополнительные инструменты.**
- `provider` — управление состоянием и DI.
- `dartz` — функциональные типы (`Either<Failure, T>`) для обработки ошибок.
- `zxcvbn`, `password_strength` — оценка надёжности пароля.
- `file_picker`, `path_provider` — файловые операции.
- `qr_flutter`, `mobile_scanner` — генерация и сканирование QR-кодов.
- `local_auth` — биометрическая аутентификация.
- `build_runner`, `mockito` — генерация моков для тестов.
- `flutter_lints` — статический анализ.
- `flutter_launcher_icons` — генерация иконок.
- Git + GitHub — система контроля версий.

**2.3.8. Архитектурное решение.**
Clean Architecture в трёх слоях + общие модули (см. Component-диаграмму в Приложении Б):

- `presentation/` — пользовательский интерфейс (экраны, контроллеры, виджеты).
- `domain/` — бизнес-логика (сущности, use case, абстракции репозиториев, валидаторы, доменные сервисы).
- `data/` — реализация репозиториев (модели, источники данных, работа с БД, криптография).
- `core/` — общие утилиты, константы, ошибки, сервисы.
- `shared/` — переиспользуемые UI-компоненты.

Обоснование выбора: инверсия зависимостей обеспечивает независимое тестирование слоёв и возможность замены СУБД без переписывания бизнес-логики.

**2.3.9. Схема взаимодействия компонентов.**
Краткое описание двух сквозных сценариев (полные sequence-диаграммы — в Приложении Б):

- *Сценарий 1.* Генерация пароля: `GeneratorScreen` → `GeneratorController` → `GeneratePasswordUseCase` → `PasswordGeneratorRepository` → `PasswordGeneratorLocalDataSource` → `Random.secure()` + оценка `zxcvbn` → ответ.
- *Сценарий 2.* Разблокировка по PIN с биометрическим резервным каналом: `AuthScreen` → `PinInputWidget` → `AuthController` → `VerifyPinUseCase` → `AuthRepository` → `AuthLocalDataSource` (PBKDF2 + constant-time сравнение) → при неудаче предложение биометрии через `local_auth` → ответ.

**2.4.3. Реализация бизнес-логики.**
Раздел строится по модулям. Для каждого — описание в тексте + ссылка на полный листинг в Приложении Д.

1. **Модуль аутентификации:** PIN с PBKDF2 (600 000 итераций), constant-time сравнение, блокировка после 5 неудачных попыток, биометрия через `local_auth` как второй фактор, многопрофильность.
2. **Модуль генерации паролей:** CSPRNG (`Random.secure()`), настраиваемые параметры (длина, наборы символов, уникальность, исключения), оценка надёжности через `zxcvbn`, сохранение конфигурации для восстановления.
3. **Модуль шифрования:** PBKDF2 → ChaCha20-Poly1305 (nonce 12 байт), версионирование параметров шифрования (EncryptionParams v1 / v2), затирание ключей после использования.
4. **Модуль глитчирования:** детерминированное преобразование входного текста по фиксированным правилам (смещения, подстановка визуально схожих символов, соление) — описание алгоритма в главе + полный листинг в приложении.
5. **Модуль хранилища:** CRUD паролей, работа с историей, категориями, поиск и фильтрация.
6. **Модуль импорта/экспорта:** формат `.passgen` (заголовок + версия + флаги + nonce + длина + зашифрованные данные + MAC), QR-обмен отдельными записями.
7. **Модуль уведомлений:** выявление слабых, устаревших и дублирующихся паролей (`PasswordStrengthNotificationService`).
8. **Модуль журнала безопасности:** запись событий с указанием типа, времени, деталей.

**2.4.5. Обеспечение безопасности информации.**

*Криптографические алгоритмы* (сводная таблица):
